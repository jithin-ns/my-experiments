Production Go-Live Runbook

Application: React + Python Platform

Release: Initial Production Launch (v1.0)

---

1. Objective

This document outlines the step-by-step process for deploying the application to production for the first time and making it available for live users.

---

2. Pre-Go-Live Preparation

2.1 Release Finalization

- Confirm production release branch
- Tag release version (v1.0)

2.2 Production Configuration

- Update GitHub Environment Variables
- Verify:
  - API endpoints
  - Database connection strings
  - Authentication configs
  - Storage configs
  - Email / Notification configs

2.3 Infrastructure Readiness

Ensure the following are provisioned and verified:

- Backend hosting environment
- Frontend hosting environment
- Database instance
- Storage (if applicable)
- Domain mapped
- SSL certificates active
- CDN configured (if used)

---

3. Deployment Steps

Step 1: Backend Deployment

- Build backend artifact / container
- Deploy Python services to production environment
- Verify:
  - Service startup success
  - Health check endpoint responding

Step 2: Database Setup

- Run initial schema setup / migrations
- Seed essential master data (if required)
- Validate DB connectivity from backend

Step 3: Frontend Deployment

- Build React production bundle
- Deploy to hosting platform
- Connect frontend to production backend

Step 4: CDN Cache Invalidation

- Clear CDN cache (if applicable)
- Ensure latest UI is served

---

4. Post Deployment Validation

4.1 Smoke Testing

Validate:

- Application loads successfully
- User login works
- Core workflows function
- API calls succeed
- Data creation & retrieval works

4.2 Sanity Testing

- End-to-end functional validation
- UI responsiveness check
- Error handling validation

---

5. Go-Live Enablement

Step 1: Enable User Access

- Disable any access restrictions / maintenance settings
- Activate production authentication

Step 2: Stakeholder Confirmation

- Functional validation sign-off

---

6. Monitoring

Immediately after go-live:

- Verify application logs
- Monitor API responses
- Check error rates
- Validate performance baseline

---

7. Rollback Readiness

Ensure availability of:

- Previous backend build
- Previous frontend build
- Database rollback strategy (if needed)

Rollback triggers may include:

- Critical functionality failure
- Data inconsistency
- System instability

---

8. Post Go-Live Hypercare (First 24–48 hrs)

Monitor:

- Login success rate
- API latency
- Error logs
- User-reported issues

---

9. Go-Live Completion Criteria

Go-live is considered successful when:

- All smoke tests pass
- Core workflows function correctly
- No critical errors observed
- Stakeholder approval received
